  PILLAR 1 — Problem Understanding & Business Relevance (20%)
  ────────────────────────────────────────────────────────────
  Criteria:
    • Problem articulation & business impact
    • Solution scope, differentiation & innovation

  Our story:
    Banks process millions of transactions daily. Laundering hides not in
    individual transactions but in the GRAPH STRUCTURE — rings of accounts
    that move money in patterns (fan-out, cycles, layering chains). Traditional
    rule-based systems flag single transactions. Argus detects the ring.

    Business impact:
    - AML compliance failures cost banks billions in fines (HSBC $1.9B,
      Standard Chartered $1.1B)
    - Analyst review time is the bottleneck — Argus surfaces analyst-ready
      alerts with explainability, not raw transaction dumps
    - Recall-tuned (catching launderers > analyst time) but with human oversight

    Differentiation:
    - Graph Neural Network on the TRANSACTION GRAPH (not tabular features)
    - 8 AML pattern types auto-classified (FAN_OUT, CYCLE, STACK, etc.)
    - GNNExplainer shows WHY each transaction was flagged (edge importance)
    - Full analyst workflow: investigate → decide → audit trail
    - Not a black box — every alert has a reason

  Talking points for judges:
    "Traditional AML catches 1 suspicious transaction. Argus catches the
     laundering ring behind 50 transactions — and shows you which edges
     are most suspicious and why."

  2a. Technical Architecture & Data Flow
  ───────────────────────────────────────
  Stack:
    ML:       PyTorch + PyTorch Geometric (GATConv / NNConv), NetworkX
    Backend:  FastAPI (Python), SQLite (argus.db), threading + RLock
    Frontend: Vanilla JS, Cytoscape.js (graph), Chart.js (charts)
    Deploy:   Render (web service), uvicorn

  Data flow (be ready to explain this end-to-end):
    1. CSV transaction data → build_graph() → PyG HeteroData
    2. MultiGNN.forward() → per-edge laundering probability (0–1)
    3. GNNExplainer → per-edge importance score
    4. Threshold filter → flagged edges → NetworkX clustering
    5. Topology classifier → FAN_OUT / CYCLE / STACK / etc.
    6. serialize_alerts() → JSON alert shape
    7. SQLite persistence + in-memory ALERTS dict
    8. FastAPI REST endpoints → Frontend via fetch()
    9. Cytoscape.js renders graph, Chart.js renders dashboard

  Engineering decisions worth mentioning:
    - Alert IDs are SHA-1 hash of sorted node set → stable across re-scans,
      analyst decisions survive model swaps
    - Pipeline cache keyed to model mtime → auto-invalidates on model swap
    - Decision audit log is append-only (no UPDATE) → full history preserved
    - Background thread for pipeline → server stays responsive during inference
    - Rate limiting (slowapi) on all endpoints
    - Whitelist/exemption system → reduces false positives for known entities

  2b. AI/ML Usage & Model Justification
  ──────────────────────────────────────
  Why GNN and not XGBoost/Random Forest?
    - AML laundering is a GRAPH problem. Features like "amount" or "frequency"
      miss the structural pattern. A cycle only looks suspicious when you see
      all 5 accounts together — a tabular model sees each row independently.
    - GNNs propagate information across the transaction graph — each account's
      embedding encodes its neighborhood's behavior.

  Why edge-level classification?
    - We classify TRANSACTIONS (edges), not accounts (nodes). This lets us
      pinpoint exactly which transfers are suspicious, not just flag an account.

  GNNExplainer integration:
    - After scoring, we run GNNExplainer on flagged transactions to produce
      an importance score per edge (0–1). This directly answers "why flagged."
    - Visualized in the graph as edge color (gray → orange → red) and timeline
      dot glow effects. High-importance edges highlighted on hover.

  Model metrics (current deployed):
    - AUC: 0.97–0.98 (the story: graph structure is highly discriminative)
    - F1: ~0.45 (threshold tuned for recall — catching launderers is priority)
    - Trained on IBM AML datasets (HI-Small, HI-Medium)
    - Recall-tuned: missing a launderer >> reviewing a clean transaction

  3c. Q&A Prep — Likely Judge Questions
  ──────────────────────────────────────
  Q: Why not use a simpler model like XGBoost?
  A: "XGBoost sees one transaction at a time. Money laundering is a graph
     crime — it only looks suspicious when you see the RING. A GNN propagates
     neighborhood information: this account's embedding reflects who it
     transacts with, not just its own behavior."

  Q: What's your AUC / F1? Isn't 0.45 F1 low?
  A: "F1 depends on threshold. We tune for recall — our priority is catching
     launderers, not minimizing analyst work. The 0.97 AUC is the real
     metric: our model ranks suspicious transactions correctly 97% of the time.
     The analyst workflow handles the precision side."

  Q: How does the explainability work?
  A: "We run GNNExplainer post-inference on each flagged cluster. It learns
     a soft mask over edges that best explains the model's prediction —
     essentially asking 'which edges matter most for this score?' That mask
     becomes the importance score shown as edge color in the graph."

  Q: How does this scale to production?
  A: "The current SQLite + in-memory architecture handles demo scale.
     The code has a PostgreSQL migration path documented — swap SQLite for
     asyncpg, replace the file cache with Redis, and the API layer is
     unchanged. The ML pipeline is the bottleneck — that moves to a message
     queue (Celery/Kafka) in production."

  Q: What data did you train on?
  A: "IBM AML datasets — synthetic but realistic, generated from the same
     family of laundering pattern simulators used in academic AML research.
     HI-Small and HI-Medium variants. The graph has 180K+ nodes and
     550K+ edges."

  Q: How do you handle false positives?
  A: "Three layers: the whitelist/exemption system suppresses known entities
     (central banks, large institutions). The analyst decision system lets
     compliance officers dismiss alerts with a reason. And suppressed alerts
     are never deleted — they're visible in the suppressed table for audit."

  Q: Is this deployable today?
  A: "Yes — it runs as a single FastAPI web service. Start command is one
     line: uvicorn src.backend.api.main:app. We have a Render deployment
     configured. The only external dependency is the CSV transaction data
     and the trained model weights."
